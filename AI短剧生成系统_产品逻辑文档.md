# AI 短剧生成系统 — 产品逻辑文档

> 本文档描述一个 AI 驱动的短剧视频生产系统的完整运作逻辑，不绑定具体技术栈和 AI 服务商。
> 用途：作为从零复刻该系统的开发指导手册。

---

## 一、系统概述

### 1.1 产品定位

用户输入一个故事创意，系统自动完成从剧本到成片的全部流程：**剧本生成 → 角色设计 → 分镜生成 → 视频生产 → 字幕合成 → 最终成片**。

核心理念：**AI 负责创作，人类负责审核把关**——每个关键节点都会暂停等待用户确认。

### 1.2 整体流程总览

```
用户输入故事创意
       │
       ▼
  剧本大纲生成 ──→ 用户审核
       │
       ▼
  分章剧集生成 ──→ 用户审核
       │
       ▼
  角色提取 ──→ 用户审核
       │
       ▼
  角色图片生成 ──→ 用户审核 & 锁定
       │
       ▼
  九宫格分镜生成 ──→ 用户融合参考图
       │
       ▼
  视频提示词构建
       │
       ▼
  视频生成（逐镜）
       │
       ▼
  字幕生成
       │
       ▼
  视频合成（拼接 + 字幕烧录）
       │
       ▼
  最终成片
```

### 1.3 需要的 AI 能力（不绑定具体服务商）

| 能力 | 用途 |
|------|------|
| 文本生成（LLM） | 剧本生成、角色提取、场景分析、提示词构建、提示词增强 |
| 图像生成 | 角色图片（表情图、三视图）、九宫格分镜、图片高清修复 |
| 视频生成 | 分镜视频片段生成（文生视频、图生视频） |
| 对象存储 | 所有图片、视频、字幕文件的存储与公网访问 |

### 1.4 需要的工具能力

| 工具 | 用途 |
|------|------|
| 视频处理工具（如 FFmpeg） | 视频片段拼接、字幕烧录、帧提取 |

## 二、项目状态机

项目有一个统一的状态机驱动整个流程流转。前端根据当前状态决定显示哪个页面。

### 2.1 状态流转图

```
[创建项目]
    │
    ▼
  DRAFT（草稿）
    │
    ▼
  OUTLINE_GENERATING（大纲生成中）
    │
    ├── 成功 → OUTLINE_REVIEW（大纲审核）
    │              │
    │              ├── 生成剧集 → EPISODE_GENERATING（剧集生成中）
    │              └── 用户要求修改 → 回到 OUTLINE_GENERATING
    │
    └── 失败 → OUTLINE_GENERATING_FAILED → 可重试
                    │
                    ▼
  EPISODE_GENERATING（剧集生成中）
    │
    ├── 成功 → SCRIPT_REVIEW（剧本审核）
    │              │
    │              ├── 生成下一章 → EPISODE_GENERATING（循环，必须按顺序）
    │              ├── 用户确认（所有章节生成完毕后）→ SCRIPT_CONFIRMED
    │              └── 用户要求修改 → 回到 EPISODE_GENERATING
    │
    └── 失败 → EPISODE_GENERATING_FAILED → 可重试
                    │
                    ▼
  SCRIPT_CONFIRMED（剧本已确认）
    │
    ▼
  CHARACTER_EXTRACTING（角色提取中）
    │
    ├── 成功 → CHARACTER_REVIEW（角色审核）
    │              │
    │              └── 用户确认 → CHARACTER_CONFIRMED
    │
    └── 失败 → CHARACTER_EXTRACTING_FAILED → 可重试
                    │
                    ▼
  CHARACTER_CONFIRMED（角色已确认）
    │
    ▼
  IMAGE_GENERATING（角色图片生成中）
    │
    ├── 成功 → IMAGE_REVIEW（图片审核）
    │              │
    │              └── 用户确认 → ASSET_LOCKED（素材锁定）
    │
    └── 失败 → IMAGE_GENERATING_FAILED → 可重试
                    │
                    ▼
  ASSET_LOCKED（素材已锁定）
    │
    ▼
  STORYBOARD_GENERATING（分镜生成中）        ← 每集独立执行
    │
    ├── 成功 → STORYBOARD_REVIEW（分镜审核）
    │              │
    │              └── 用户确认 → 进入视频生产管线
    │
    └── 失败 → STORYBOARD_GENERATING_FAILED → 可重试
                    │
                    ▼
  PRODUCING（视频生产中）                    ← 每集独立执行，内部有子阶段（见第六章）
    │
    └── 完成 → COMPLETED（生产完成）
```

### 2.2 状态机设计要点

- 每个状态映射到前端的步骤编号（frontendStep），前端据此决定显示哪个页面
- 所有 `*_FAILED` 状态都允许用户点击重试，回到对应的 `*_GENERATING` 状态
- 系统只支持多集模式，不支持单集模式
- 章节生成必须按顺序进行，不能跳章（`validateChapterOrder`）
- `SCRIPT_CONFIRMED` 的前置条件：所有章节的剧集都已生成完毕
- `ASSET_LOCKED` 表示角色图片已确认不可再修改，此后进入分镜生成和视频生产流程
- `STORYBOARD_GENERATING` 和 `PRODUCING` 以**单集为单位**独立执行
- `PRODUCING` 阶段内部有子阶段（见第六章）

## 三、剧本生成系统

### 3.1 剧本大纲生成

**目的**：根据用户的故事创意，生成结构化的故事大纲。

**输入**：
- 故事描述（用户的一句话创意）
- 类型（genre，如悬疑、爱情、奇幻）
- 目标受众（targetAudience）
- 总集数（totalEpisodes）
- 单集时长（episodeDuration）
- 视觉风格（visualStyle）

**处理逻辑**：
1. 构建提示词，注入世界观规则和格式要求
2. 调用 LLM 生成 Markdown 格式的大纲
3. 保存大纲到项目

**输出**：Markdown 格式大纲，包含：
- 角色定义（主角 / 反派 / 配角）—— **这是后续角色提取的唯一数据来源**
- 道具定义（核心道具 / 辅助道具 / 世界观道具）
- 章节结构（每章 2~5 集）

**用户操作**：审核大纲 → 可修改 / 重新生成 → 确认

### 3.2 分章剧集生成

**目的**：按章节顺序生成每一集的详细内容。所有章节必须全部生成后才能确认剧本。

**输入**：
- 用户指定要生成的章节标题（chapter 参数）
- 已确认的大纲（作为上下文）
- 已生成的剧集（携带前序摘要，保持剧情连贯性）

**处理逻辑**：
1. `validateChapterOrder` 验证章节顺序——必须按顺序生成，不能跳章
2. 构建提示词，携带大纲上下文和已生成剧集的摘要
3. LLM 逐集生成 JSON 格式的剧集内容
4. 保存剧集到 Episode 表
5. 状态流转到 `SCRIPT_REVIEW`，用户可继续生成下一章或确认剧本

**输出**：每集的 JSON 数据，包含：
- 标题、内容
- 出场角色列表
- 出场道具列表
- 剧情连续性维护信息

**用户操作**：审核剧集 → 生成下一章 / 修改当前章 / 确认剧本（所有章节生成完毕后才可确认）

## 四、角色管理系统

### 4.1 角色提取

**目的**：将大纲中已定义的角色信息格式化为结构化数据，供后续角色图片生成和视频提示词构建使用。

**输入**：项目大纲（`project.scriptOutline`）+ 故事创意（`project.storyPrompt`）

**处理逻辑**：
1. 读取项目大纲（大纲阶段已包含完整的角色定义，是角色的唯一来源）
2. 构建 prompt，让 LLM 将大纲中的角色定义解析为标准化的 JSON 数组
3. 解析 AI 返回的 JSON，提取每个角色的结构化信息
4. 先删除旧角色数据，再保存新角色数据到 Character 表

**输出**：角色实体列表（`List<CharacterDraftModel>`），每个角色包含：
- 基础信息：名称（name）、角色定位（role）：主角 / 反派 / 配角
- 外貌描述（appearance）
- 性格特征（personality）
- 背景故事（background）
- 视觉风格（visualStyle）：从项目设置继承

**用户操作**：审核角色列表 → 确认

### 4.2 角色图片生成（三步法）

角色图片生成采用三步递进策略，确保角色视觉形象的一致性。

#### 第一步：生成角色档案

为每个角色生成一个 `appearancePrompt`（外观描述词）——专门给图像生成模型看的、精确描述角色外观的英文提示词。

#### 第二步：生成表情九宫格

用角色的外观描述词，让图像生成 AI 在一张图上同时画出角色的 **9 种表情**（喜悦、愤怒、悲伤、惊讶、恐惧、厌恶、中性、思考、疲惫）。

关键设计：
- **只画头部和肩部**，不画全身——头像一致性比全身更容易控制
- **纯色背景**——避免背景干扰 AI 对角色的理解
- **3×3 网格构图**——利用 AI 的同画布一致性本能（多个画面在同一张图上生成时，AI 会本能地保持风格统一）
- **文字检测**：生成后检查图片里是否意外出现了文字（AI 有时会在图片里写标签），如果检测到文字就重新生成，最多重试 3 次

#### 第三步：生成三视图

以表情图为参考，让 AI 画出角色的**正面、侧面、背面**全身图。

关键设计：
- **传入表情图作为参考图片**——这是让三视图和表情图保持同一张脸的核心手段
- **全身站立、中性表情、纯色背景**——标准化姿态，方便下游使用
- **垂直排列、对齐一致**——三个视图的比例和位置严格对齐

### 4.3 角色一致性保障机制

#### 下游使用优先级

当视频生成需要角色参考图时，优先级为：**三视图 > 表情图**

原因：三视图包含的信息更完整（全身 + 多角度），能帮助视频模型更准确地理解角色的整体外观。

#### 负面提示词

不同视觉风格的角色生成，系统会附带不同的"排除清单"。以二次元风格为例：
- 排除不安全内容、文字、水印
- 排除不好的解剖结构
- 排除写实风格、照片风格、3D 风格
- 排除皮肤纹理、毛孔（真人风格特征）

### 4.4 素材锁定

用户确认所有角色图片后，进入 `ASSET_LOCKED` 状态，角色图片不可再修改。此后进入视频生产流程。

## 五、分镜系统

### 5.0 分镜生成（StoryboardService）

**目的**：为一集剧本生成结构化的分镜 JSON 数据（panels），作为后续场景分析、九宫格生成、视频提示词构建的基础数据。

**触发时机**：素材锁定（`ASSET_LOCKED`）之后，视频生产（`PRODUCING`）之前。以**单集为单位**独立执行。

**输入**：
- 剧集内容（`episode.content`）
- 世界观配置
- 角色状态

**处理逻辑**：
1. 加载剧集信息和世界观配置
2. 构建系统提示词和用户提示词（`PromptBuilder.buildStoryboardSystemPrompt`）
3. 调用 AI 文本生成服务
4. 解析并验证返回的 JSON 结果
5. 保存到 `episode.storyboardJson` 字段
6. 更新角色状态

**输出**：`storyboardJson` 字段，结构如下：

```json
{
  "episode": 1,
  "title": "集标题",
  "panels": [
    {
      "panel_id": "ep1_p1",
      "shot_type": "WIDE_SHOT",
      "camera_angle": "eye_level",
      "composition": "构图描述",
      "background": {
        "scene_desc": "场景",
        "time_of_day": "时段",
        "atmosphere": "氛围"
      },
      "characters": [
        {
          "char_id": "角色ID",
          "position": "位置",
          "pose": "姿势",
          "expression": "表情",
          "costume_state": "normal"
        }
      ],
      "dialogue": [
        {
          "speaker": "char_id 或 narrator",
          "text": "内容",
          "bubble_type": "speech"
        }
      ],
      "sfx": ["音效"],
      "pacing": "normal",
      "image_prompt_hint": "给图像AI的视觉提示"
    }
  ]
}
```

**前置条件**：`episode.getStoryboardJson()` 为空时才能生成。

### 5.1 场景分析

**目的**：将一集的分镜数据（`storyboardJson` 中的 panels）拆分成若干「场景组」，用于后续统一生成参考图。

**输入**：一集的 `storyboardJson`（由 5.0 分镜生成步骤产出）

**处理逻辑**：解析 panels 数组，按场景（scene）对分镜进行分组

**输出**：场景组列表（`List<SceneGroupModel>`），每个组包含：
- `sceneId`：场景组标识
- `description`：场景描述
- `characters`：关联角色列表
- `startPanelIndex` / `endPanelIndex`：关联分镜索引范围

**为什么分组**：同一场景组的分镜共享参考图，保证角色和场景的一致性。

### 5.2 九宫格分镜生成

#### 为什么是九宫格？

这是整个系统最关键的设计决策之一。

当图像生成 AI 在**同一张画布上**同时画出 9 个画面时，它会本能地保持这 9 个画面之间的风格一致——因为它们共享同一个生成上下文。

如果分别画 9 张图（9 个独立请求），AI 每次都会"重新理解"需求，导致：
- 同一个角色在 9 张图里长得不一样
- 画风在 9 张图之间飘忽不定
- 光线、色调、构图无法统一

所以系统策略是：**让 AI 一次性在一张大图上画出 3×3 的九宫格分镜**，然后切成 9 张独立的图片。

#### 九宫格生成

**输入**：场景组（场景描述 + 关联角色）

**处理逻辑**：
1. 构建图像生成提示词，注入角色外观描述 + 场景描述
2. 调用图像生成 AI 生成九宫格图
3. 上传至对象存储
4. 每生成一张就保存到生产记录，供前端实时展示

**输出**：九宫格图片 URL

### 5.3 九宫格拆分

#### 拆分原理：数组下标对齐（非图像识别）

拆分的核心是**数组下标对齐**：

**输入**：
1. 一张九宫格大图（图片文件）
2. 一个包含 9 个分镜描述的数组（文字数据），每个元素包含对应分镜的全部元数据

**拆分过程**：

```
九宫格大图（3行 × 3列）
┌─────┬─────┬─────┐
│  0  │  1  │  2  │    ← 第 0 行
├─────┼─────┼─────┤
│  3  │  4  │  5  │    ← 第 1 行
├─────┼─────┼─────┤
│  6  │  7  │  8  │    ← 第 2 行
└─────┴─────┴─────┘

分镜数据数组：
[
  { 场景: "花园", 景别: "全景", ... },   ← 下标 0
  { 场景: "室内", 景别: "中景", ... },   ← 下标 1
  ...
  { 场景: "山顶", 景别: "大远景", ... }, ← 下标 8
]
```

系统按照**从左到右、从上到下**的顺序（行优先）切割，每个位置的小图与数组中相同下标的分镜数据绑定。

**切割实现**：
1. 在内存中创建一个与单格等大的画布
2. 把大图上对应区域裁剪下来画到小画布上
3. 把小画布导出为图片数据

切割完成后，每一张小图都被"绑"上了对应分镜的全部文字描述。

#### 六宫格支持

系统同时支持 3×3（九宫格）和 2×3（六宫格）两种布局，切割逻辑相同，只是行列数不同。

#### 多页分镜处理

分镜按**场景组**独立分页，而非按整集分页：

- 每个场景组独立生成九宫格
- 场景组内的分镜数不足 9 个时，剩余格子用**纯色填充**作为废弃格子
- 场景组内的分镜数超过 9 个时，该场景组分多页处理

示例：
```
场景组A（7个分镜）→ 1页九宫格，最后2格纯色填充
场景组B（12个分镜）→ 第1页9格 + 第2页3格 + 6格纯色填充
场景组C（3个分镜）→ 1页九宫格，6格纯色填充
```

计算公式：`起始下标 = 页码 × 每格数量`（每个场景组独立计算）

**容错设计**：某一页切割失败不会中断整个流程，系统跳过该页继续处理下一页。

### 5.4 分镜增强

系统支持两种方式为分镜补充专业的摄影参数。

#### 智能推荐（基于规则，不消耗 AI）

内置推荐引擎，根据分镜描述文字自动推荐景别、角度和运镜：
- 描述中出现"脸"或"表情"→ 推荐特写
- 描述中出现"跟随"或"行走"→ 推荐正跟随运镜
- 描述中出现"俯拍"或"高位"→ 推荐高位俯拍角度

#### AI 增强（调用 LLM 精选参数）

LLM 分析每个分镜的场景描述，从预定义的专业术语库中选择最合适的组合：
- 8 种景别（大远景 → 细节大特写）
- 6 种角度（视平 → 鸟瞰）
- 11 种运镜（固定 → 滑轨横移）

AI 需要给出选择理由。系统会验证返回值是否在标准术语库中，如果不在，自动回退到默认值（中景 + 视平 + 固定）。

**防限流**：批量增强多个分镜时，每个分镜之间插入 0.5 秒延迟。

## 六、视频生产系统

视频生产是系统最核心也最复杂的阶段，以**单集**为单位执行，内部包含 7 个子阶段。

### 6.1 单集生产管线

```
启动单集生产
    │
    ▼
阶段0: 分镜生成（StoryboardService，生成 panels JSON）
    │
    ▼
阶段1: 场景分析（解析 panels，按场景分组）
    │
    ▼
阶段2: 九宫格生成（每个场景组生成一张参考网格图）
    │
    ▼
阶段3: 用户融合（人工介入 — 在分镜底图上叠加角色参考图）
    │
    ▼
阶段4: 视频提示词构建
    │
    ▼
阶段5: 视频生成（逐镜生成视频片段）
    │
    ▼
阶段6: 字幕生成
    │
    ▼
阶段7: 视频合成（拼接 + 字幕烧录）
    │
    ▼
最终视频
```

### 6.2 阶段 3：用户融合（关键人工节点）

这是整个流程中**唯一需要用户手动操作的环节**。详见第七章。

### 6.3 阶段 4：视频提示词构建（VideoPromptBuilderService）

**目的**：将分镜数据翻译成视频生成模型能理解的提示词格式。

**服务类**：`VideoPromptBuilderService`

#### 按 videoProvider 分支构建

系统根据配置的视频提供商（`comic.video.provider`）选择不同的构建策略：

##### Sora 模式（中文直拼）

适用于支持中文提示词的视频模型（如 Sora）：
- 直接将分镜数据格式化为中文提示词
- 格式：`第N镜 / 时长 / 场景：景别，角度，运镜，描述，对白，特效`
- 不需要调用 LLM

输出格式示例：
```
第1镜 / 5秒 / 场景：大远景，高位俯拍，固定，美丽的花园场景，阳光明媚
第2镜 / 5秒 / 场景：中景，视平，横移，室内场景，角色A正在看书，对白：你好
```

##### 通用模式（LLM 中翻英 + 降级兜底）

适用于大多数视频生成模型（如 Seedance）：

1. **LLM 翻译**：将中文分镜描述翻译为英文视频提示词
   - 构建包含风格前缀、景别、角度、运镜、场景描述、对白、特效的完整 prompt
   - 注入角色一致性指令（从 Character 表读取角色外貌描述）
   - 调用 `TextGenerationService.generate()` 执行翻译

2. **降级兜底**（`buildFallbackEnglishPrompt`）：如果 LLM 调用失败，直接用中英映射表拼接基础英文提示词

#### 风格前缀（buildStylePrefix）

系统根据视觉风格（`visualStyle`）生成英文风格前缀，作为提示词的开头：

| 风格关键词 | 风格前缀核心 |
|-----------|-------------|
| anime/动漫/日系 | Anime style, Japanese animation, cel-shaded, vibrant colors, clean linework... |
| real/写实/真人 | Photorealistic style, cinematic live-action quality, natural lighting... |
| 3d/三维 | High quality 3D animation style, Pixar-quality rendering, subsurface scattering... |
| manga/漫画 | Manga style, black and white with screen tones, bold linework... |
| watercolor/水彩 | Watercolor painting style, soft blending of colors, visible brushstrokes... |
| oil/油画 | Oil painting style, rich impasto texture, visible brushwork... |
| pixel/像素 | Pixel art style, 16-bit retro game aesthetic... |

#### 角色一致性指令（buildCharacterConsistencyInstruction）

为每个出场角色注入外貌描述，确保视频生成时角色外观与参考图一致：
```
CRITICAL CHARACTER CONSISTENCY REQUIREMENTS:
- All characters must EXACTLY match the provided reference image in appearance
- Character 角色名: 外貌描述
```

#### 任务分组（VideoTaskGroupModel）

多个分镜按场景组（`SceneGroupModel`）和时长上限（`maxDurationPerGroup`，默认 10 秒）分组为 `VideoTaskGroupModel`，每组对应一次视频生成请求。

### 6.4 阶段 5：视频生成

#### 统一平台层

系统通过统一的视频生成代理层，支持接入多个视频生成模型。用户不需要知道每个模型的 API 格式区别，系统自动适配。

#### 提交任务

系统把以下信息打包发送给视频生成平台：
- 提示词（上一阶段构建好的）
- 参考图片地址（如果开启了图片融合）
- 模型参数：宽高比（16:9 横屏 / 9:16 竖屏）、时长、画质

不同模型有些特殊处理：
- 有的模型额外需要横竖屏方向参数
- 有的模型开启提示词增强（自动翻译为英文）和自动放大
- 部分模型接口格式差异大，需要单独的提交逻辑

#### 轮询等待机制

视频生成不是即时完成的，采用轮询机制：
1. 提交任务后拿到任务编号
2. 每 5 秒向平台查询一次任务状态
3. 状态有四种：排队中 → 处理中 → 已完成 / 失败
4. 已完成 → 提取视频下载地址
5. 失败 → 抛出错误

**超时保护**：默认最多轮询 120 次（约 10 分钟），用户可手动设置。

**取消机制**：用户可在等待过程中取消。取消后只停止本地轮询，视频平台可能仍在生成但系统不再关心。

#### 并发控制

信号量（`Semaphore(1)`）限制同时只能有 1 个视频生成任务（资源密集型），多个任务按队列依次执行。

> **TODO**：当前信号量为全局单例，所有用户共享 1 个并发槽位。未来多用户场景需改为按用户/项目隔离。

#### 进度显示

进度按图片生成和视频生成的**完成百分比**计算，不采用分段映射。前端轮询获取当前阶段的完成情况并显示进度条。

### 6.5 镜头间连续性（同集内）

系统支持在同一集内实现镜头间的视觉连续性，通过 Seedance API 的首尾帧模式实现。

**基础设施**（已实现）：
- `VideoProductionTask` 实体有 `lastFrameUrl` 字段
- `SeedanceVideoService.generateVideo()` 支持 `firstFrameUrl` + `lastFrameUrl` 参数
- API 返回的 `TaskStatus` 包含 `lastFrameUrl`（生成视频的最后一帧截图）

**串联逻辑**（TODO：待实现）：
在视频生成队列中，每段视频生成完成后，将其 `lastFrameUrl` 传给下一段视频的 `firstFrameUrl`，实现同集内镜头间的视觉衔接。

**范围**：仅在**同一集内**的镜头之间串联，不跨集。

### 6.5 五种视频生成策略

除了常规的"图生视频"和"文生视频"，系统还支持四种高级策略。

#### 策略一：常规生成

- **图生视频**：以分镜图片为输入，生成视频片段
- **文生视频**：以文本提示词为输入，直接生成视频

#### 策略二：剧情延续

**用途**：让多段视频在视觉上连贯，像连续剧一样。

**做法**：取上一段视频的**最后一帧**作为下一段视频的输入图片。

**关键细节**：系统故意**不传视频文件本身**，只传最后一帧截图。因为如果传了视频文件，有些平台会进入"视频编辑"模式，而我们要的是"用这张截图作为起点，生成一段全新的视频"。

#### 策略三：首尾帧插值

**用途**：用户指定起始画面和结束画面，AI 自动生成两帧之间的过渡动画。

**做法**：把两帧图片都传给视频生成服务，AI 会额外调用 LLM 来"编排"提示词，然后视频模型根据编排后的提示词生成过渡动画。

#### 策略四：场景切换

**用途**：从现有视频中截取一个画面，经过高清修复后，作为新场景的起点。

**做法**：
1. 从上游视频中提取一帧画面
2. 调用 AI 进行高清修复和放大——但严格保持原始构图不变
3. 用修复后的高清图作为新视频的输入

**高清修复的约束非常严格**：
- "严格保持原始构图，不要重新取景，不要改变视角"
- "不要添加新的人物"
- "只增强现有像素，不要发明新的几何形状"
- "人数必须和输入完全一致"

#### 策略五：角色迁移

**用途**：保持角色外观，但替换场景或动作。

**做法**：
1. 用 AI 分析源视频，提取**纯动作描述**（不包含人物身份信息）
2. 把动作描述和用户的目标提示词组合
3. 以角色图片作为输入，生成新视频

**为什么只提取动作不提取人物**：如果提示词里包含人物描述，AI 可能会"重新理解"人物外观，导致和参考图不一致。只描述动作可以让 AI 专注于"这个人在做什么"，角色外观由参考图决定。

## 七、图片融合系统

### 7.1 为什么需要图片融合？

单独给视频模型一张分镜小图时，视频模型只知道"这一个画面"的信息。通过在分镜底图上叠加角色参考图，可以让视频模型同时理解场景构图和角色外观。

> **TODO**：当前融合方案为临时实现，最终方案待确定。以下是待评估的改进方向：

### 7.1.1 融合方案改进方向

融合要解决的核心问题：**怎么给视频模型提供最好的参考素材，让它生成的视频角色长得一致、风格统一。**

#### 方案 A：只传角色参考图
- 视频模型拿到角色三视图/表情图 + 文字描述场景
- 优点：角色外观最准确
- 缺点：视频模型不知道场景长什么样
- 前提：视频模型能接受独立的角色参考图

#### 方案 B：只传分镜底图
- 视频模型拿到九宫格切出来的分镜小图 + 文字提示词
- 优点：场景构图准确
- 缺点：角色可能跟参考图长得不一致

#### 方案 C：逐格叠加融合（当前实现）
- 在分镜底图上叠加角色图，合成一张图传给视频模型
- 优点：角色 + 场景都有
- 缺点：叠加质量取决于用户手动调整，合成图边缘/光影可能不自然

#### 方案 D：多图参考（如果视频模型 API 支持）
- 同时传角色图和分镜图，不合成，让模型自己理解
- 优点：最干净，没有合成伪影
- 缺点：取决于 Seedance 等 API 是否支持多图输入

**决策依据**：先确认当前使用的视频生成平台（Seedance）是否支持多张参考图输入。如果支持，方案 D 最优；如果不支持，在方案 A/B/C 中选择或组合。

### 7.2 逐格叠加融合（当前实现）

**核心思路**：在每个九宫格子的分镜底图上叠加角色参考图，用户手动调整叠加效果。

**前端交互流程**（GridFusionEditor 组件）：

1. **选择面板**：点击九宫格中的任意格子
2. **选择角色**：从角色面板（CharacterPalette）选择要叠加的角色
3. **选择角色图类型**：
   - 标准（standard）：标准角色肖像图
   - 三视图（threeView）：三视图网格图
   - 表情（expression）：表情网格图
4. **调整叠加参数**：
   - 拖拽调整角色位置
   - 滑块调整透明度（0-100%）
   - 滑块调整缩放比例（0.1x-2x）
5. **移除覆盖**：可移除已分配的角色
6. **翻页操作**：多页网格时支持上一页/下一页切换

**提交流程**：

1. 用户完成编辑后，前端**逐格独立生成融合图**（canvas → file）
2. 每格单独上传到 OSS，获取 URL
3. 每页提交 9 个融合图 URL 的数组（`submitFusionPage` 接口）
4. 后端保存到 `fusedGridUrls`（二维数组结构）
5. 所有页全部提交后，自动触发 `continueProductionFlow` 恢复视频生产管线

**融合图与视频生成的关联**：

后端将融合图 URL 注入到 `VideoPromptModel` 中：
- 场景组级别：`group.fusedReferenceImageUrl` = 该页第 1 格的融合图
- 镜头级别：`prompt.fusedReferenceImageUrl` = 该镜头对应格子的融合图

### 7.3 融合图上传

融合后的图片上传到 OSS，获取公网 URL。

上传失败时，Base64 数据直接作为备用方案传下去（部分平台支持直接传图片数据）。

## 八、字幕生成与视频合成

### 8.1 字幕生成

**目的**：根据分镜数据中的对话 / 旁白生成字幕文件。

**输入**：分镜数据 + 视频片段时长

**处理逻辑**：根据分镜的对白内容和对应的视频时间区间，生成标准字幕文件（如 SRT 格式）。

**输出**：字幕文件（上传至对象存储）

> **TODO（临时方案）**：当前字幕时间轴基于分镜预定义时长，可能与实际视频时长不一致。正式方案将使用 **TTS + ASR** 流程：
> 1. 对分镜对白执行 TTS（文本转语音）生成音频
> 2. 对音频执行 ASR（语音识别）获取字词级时间戳
> 3. 基于精确时间戳生成字幕文件
> 4. 音频也可合成到最终视频中

### 8.2 视频合成

**目的**：将所有视频片段 + 字幕合成为最终成片。

**输入**：
- 所有视频片段（按分镜顺序排列）
- 字幕文件

**处理逻辑**：
1. 按分镜顺序拼接所有视频片段
2. 烧录字幕到视频上
3. 上传最终视频到对象存储

**输出**：最终视频 URL

## 十、状态同步机制

### 10.1 轮询机制

由于大量操作是异步执行的，系统采用轮询机制同步状态：

```
前端                    后端                    AI 服务
  │                      │                        │
  │── 触发操作 ──────────→│                        │
  │← 返回（状态已更新）──│                        │
  │                      │                        │
  │── 轮询状态 ─────────→│                        │
  │← 当前状态 + 进度 ────│                        │
  │                      │── AI 生成请求 ────────→│
  │                      │← 生成结果 ─────────────│
  │                      │                        │
  │── 轮询状态 ─────────→│                        │
  │← 新状态（如 REVIEW）─│                        │
  │                      │                        │
  │  前端切换到审核页面   │                        │
```

**关键点**：
- 后端所有耗时操作都是异步的，接口立即返回
- 前端通过定时轮询获取最新状态
- 生产阶段会自动下沉查询剧集级别的子阶段进度

## 十一、异常处理策略

### 11.1 设计原则

- 所有失败都有明确的 FAILED 状态，不会卡在中间态
- 失败状态允许重试，用户有完整控制权
- 视频生成有自动重试机制，减少人工干预

### 11.2 分级处理

| 异常类型 | 处理策略 |
|---------|---------|
| 剧本 / 角色 / 图片生成失败 | 更新为 FAILED 状态，用户可点击重试 |
| 视频生成失败 | 自动重试最多 3 次，指数退避；仍失败则标记 FAILED，用户可触发重新生产 |
| 九宫格单页切割失败 | 跳过该页继续处理下一页（容错设计） |
| AI 增强提示词失败 | 回退到简单拼接兜底 |
| 角色图片文字检测失败 | 最多重试 3 次 |
| 融合图上传失败 | 使用 Base64 数据直接传递 |
| 分镜生成失败 | 更新为 STORYBOARD_GENERATING_FAILED，用户可重试 |

### 11.3 模型降级

AI 服务维护按优先级排列的模型列表：
1. 先尝试优先级最高的模型
2. 失败则自动切换到下一个
3. 依次尝试，直到成功或全部失败

同时适用于图像生成和文本生成服务。

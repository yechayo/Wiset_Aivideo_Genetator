① SCRIPT_PLANNER — 剧本大纲
AI 调用：文本 AI × 1 次
用户在 PROMPT_INPUT 节点输入创意（比如"一个仙侠短剧，男主离家三年后归来..."），传给 SCRIPT_PLANNER 节点。
Gemini 根据创意生成完整的剧本大纲，产出以下关键字段：

这个节点最关键的产出是 scriptVisualStyle。它决定了整部短剧的画面风格：
- "3D" → 所有提示词前缀变为 "Xianxia 3D animation character, semi-realistic style..."
- "REAL" → 变为 "Professional portrait photography, photorealistic human..."

1. 入口函数
   函数名：generateScriptPlanner
   async function generateScriptPlanner(
   prompt: string,       // 用户输入的创意文本
   config: {             // 用户在节点 UI 中填写的配置
   theme?: string,          // 主题，如 "爱情与成长"
   genre?: string,          // 类型，如 "仙侠"
   setting?: string,        // 背景，如 "古代中国，修仙世界"
   episodes?: number,       // 总集数，如 10
   duration?: number,       // 单集时长（分钟），如 1
   visualStyle?: string     // 视觉风格，"3D" / "REAL" / "ANIME"
   },
   refinedInfo?: Record<string, string[]>,  // 可选，来自剧目精炼节点的参考信息
   model?: string,                          // 可选，指定使用的 AI 模型
   context?: { nodeId?: string; nodeType?: string }  // 调用上下文（用于日志）
   ): Promise<string>    // 返回 AI 生成的 Markdown 格式剧本大纲文本
2. 第一步：动态参数计算
   根据用户设定的集数，自动计算章节数、角色数、物品数的上下限：
   const episodes = config.episodes || 10;                    // 总集数，默认10
   const episodesPerChapter = 4;                               // 每章固定4集
   const chapterCount = Math.ceil(episodes / episodesPerChapter);  // 章节数
   const minCharacters = Math.round(10 + (episodes * 0.15));  // 最少角色数
   const maxCharacters = Math.round(minCharacters * 1.3);     // 最多角色数
   const minItems = Math.round(8 + (episodes * 0.1));         // 最少物品数
   const maxItems = Math.round(minItems * 1.25);              // 最多物品数

3. 第二步：构建系统提示词（systemInstruction）
   系统提示词是一个约 200 行的固定模板，包含 8 个占位符，在这一步被替换为实际数值：
   let systemInstruction = SCRIPT_PLANNER_INSTRUCTION
   .replace(/{TotalEpisodes}/g, String(episodes))           // 总集数 → "10"
   .replace(/{ChapterCount}/g, String(chapterCount))        // 章节数 → "3"
   .replace(/{EpisodesPerChapter}/g, String(4))             // 每章集数 → "4"
   .replace(/{MinCharacters}/g, String(minCharacters))      // 最少角色 → "12"
   .replace(/{MaxCharacters}/g, String(maxCharacters))      // 最多角色 → "16"
   .replace(/{MinItems}/g, String(minItems))                // 最少物品 → "9"
   .replace(/{MaxItems}/g, String(maxItems))                // 最多物品 → "11"
   .replace(/{Duration}/g, String(config.duration || 1));   // 单集时长 → "1"
   模板的完整结构如下：
   你是一位专精于短剧和微电影的专业编剧 (Professional Screenwriter)。
   你的任务是根据用户的核心创意和约束条件，创建一个引人入胜的**中文剧本大纲**。

核心原则：剧本大纲只在章节层面规划，不细化到每集

如果提供了【参考信息 - 来自剧目精炼】：
- 这些信息仅作为创作参考，用于启发灵感和确定方向
- 不要照搬，而是将其作为创作风格、主题方向的参考
- 可以借鉴其中的情感基调、受众定位、角色特征等元素
- 最终作品应该是原创的，融合用户的核心创意和你的专业创作

## 🎯 章节结构与节奏要求

节奏规律（必须严格遵循）：

1. 小高潮：每3-5集设置一次小高潮
   第3-5集：第一次小高潮
   第8-10集：第二次小高潮
   第13-15集：第三次小高潮
   ...

2. 大转折：每10-15集设置一次大转折
   第10-15集：剧情大反转/阵营重组/重大秘密揭露
   第20-25集：中期转折/角色关系巨变

## ⚠️ 重要规则

1. 只在章节层面规划，不要细化到每集的具体场景
2. 每章100-150字，简洁描述这几集的核心内容
3. 每章必须包含明确的起承转合
4. 严格遵守节奏规律：3-5集小高潮，10-15集大转折
5. 章节之间要有因果链条，避免情节断裂
6. 角色描述按分级处理：核心角色详细，配角简单，其他一笔带过
7. 物品描述按分级处理：核心详细，辅助简略，世界物品最简
8. 物品名称必须统一，不要使用同义词
9. 单集时长参考: {Duration} 分钟
10. 在创作中请始终贯彻[Visual Style]的视觉美学
11. 如有参考信息，灵活运用但不要生搬硬套，保持原创性

9. 风格信息的传播机制
   scriptVisualStyle 不是在本节点使用的。下游节点通过一个叫 getUpstreamStyleContext 的递归查找函数，沿着节点连线向上找到 SCRIPT_PLANNER 节点，读取风格信息。
   查找逻辑（按优先级）：
1. 先在当前节点的直接输入节点中找 SCRIPT_PLANNER 类型的节点
   → 找到就直接读取它的 scriptVisualStyle / scriptGenre / scriptSetting

2. 如果没找到，在直接输入中找 SCRIPT_EPISODE 类型的节点
   → 然后从 SCRIPT_EPISODE 的输入中再找 SCRIPT_PLANNER
   → 找到就读取

3. 如果还是没找到，递归向上遍历所有输入节点
   → 直到找到 SCRIPT_PLANNER 为止
   → 这个处理的是间接连接的情况（比如中间隔了一个其他类型的节点）
   找到风格后，用另一个函数 getVisualStylePrompt 把风格枚举转成英文提示词前缀：
   "3D" → "Xianxia 3D animation character, semi-realistic style,

   PBR shading with soft translucency, subsurface scattering,
   ambient occlusion, delicate and smooth skin texture,
   flowing fabric clothing, individual hair strands,
   neutral studio lighting, clear focused gaze, natural demeanor"

"REAL" → "Professional portrait photography, photorealistic human,
cinematic photography, professional headshot,

realistic skin texture, visible pores,

- 平均镜头时长：2-3秒（保持快节奏）

- 不得出现少于1秒的碎片化镜头

**2. 分镜数量计算（必须满足最低要求）**
根据总时长智能计算分镜数量：
- **1分钟内容（60秒）**：**至少 20 个分镜**
    - 最少：20个分镜（平均3秒/镜）
    - 推荐：25-30个分镜（平均2-2.4秒/镜）
    - 最多：60个分镜（平均1秒/镜）
- **2分钟内容（120秒）**：**至少 40 个分镜**
    - 最少：40个分镜（平均3秒/镜）
    - 推荐：50-60个分镜（平均2-2.4秒/镜）
    - 最多：120个分镜（平均1秒/镜）
- **3分钟内容（180秒）**：**至少 60 个分镜**
    - 最少：60个分镜（平均3秒/镜）
    - 推荐：75-90个分镜（平均2-2.4秒/镜）
    - 最多：180个分镜（平均1秒/镜）

**3. 时间精确（强制要求）**
- **所有分镜的时长总和必须等于或大于目标总时长**
- **不得低于目标总时长**（这是底线要求）
- 如果超出，允许最多超出5秒
- 例如：目标60秒，生成总时长可以在60-65秒之间

**4. 时长不足的补偿策略**
如果计算后发现总时长不足，必须：
- 增加更多细节镜头（如：特写角色反应、环境细节）
- 将长镜头拆分为多个短镜头
- 添加过渡镜头或转场镜头
- 补充角色表情变化的镜头
- **严禁通过增加单个镜头时长来凑时间**

**5. 剧情结构智能拆分（核心要求）**

**关键情节/高潮场景**：
- 使用更多短镜头（1-2秒），快速切换，营造紧张感

**情感戏/对话场景**：
- 使用中等时长镜头（2-3秒），适度推拉

**环境描写/转场**：
- 使用较长镜头（3-4秒），建立空间感

**动作场面**：
- 使用极短镜头（1秒），快速剪辑，强化动感

**内容结构识别**
必须识别并优先处理以下结构：
- 【场景】标记：场景切换处必须有分镜点
- 【画面】标记：画面描述重点处
- 对话：每个角色发言应有独立镜头（1-2秒）
- 动作：每个关键动作分解为1-2秒镜头
- 情绪：情绪变化点应有镜头切换

6.1 切割原理
九宫格: 3列 × 3行
原始大图尺寸: 比如 1536×864

每格宽度 = 1536 / 3 = 512
每格高度 = 864 / 3 = 288
用 ctx.drawImage(img, x, y, w, h, 0, 0, w, h) 从大图里裁出每一格：
panelIndex 0: row=0, col=0 → 裁 (0, 0, 512, 288)       → 分镜1
panelIndex 1: row=0, col=1 → 裁 (512, 0, 512, 288)      → 分镜2
panelIndex 2: row=0, col=2 → 裁 (1024, 0, 512, 288)     → 分镜3
panelIndex 3: row=1, col=0 → 裁 (0, 288, 512, 288)      → 分镜4
panelIndex 4: row=1, col=1 → 裁 (512, 288, 512, 288)     → 分镜5
panelIndex 5: row=1, col=2 → 裁 (1024, 288, 512, 288)    → 分镜6
panelIndex 6: row=2, col=0 → 裁 (0, 576, 512, 288)       → 空白
panelIndex 7: row=2, col=1 → 裁 (512, 576, 512, 288)      → 空白
panelIndex 8: row=2, col=2 → 裁 (1024, 576, 512, 288)     → 空白
对应关系纯靠数组下标：九宫格的第 i 格 = storyboardShots[i]。

连续性机制
同一视频内（同一任务组）
Sora 2 原生理解 Shot 1/2/3... 格式。它会在一个视频内部：
- 理解多个 Shot 属于同一场景
- 自动处理镜头过渡
- 保持人物、环境、光照一致
  开头的 0.5 秒黑屏空镜是给 Sora 的"渐入"起点。
  跨视频（不同任务组）
  任务组之间是完全独立的 API 调用，Sora 不知道彼此的存在。靠三个"软保证"：
  保证方式
  具体机制
  可靠程度
  风格一致
  所有任务组共用同一套风格描述词（从①号节点传下来）
  较强
  角色锚定
  融合图底部带角色三视图，Sora "看到"角色长什么样
  中等
  叙事连贯
  提示词按剧本顺序编写，每个 Shot 有完整场景描述和对白
  强
  具体表现：

1. 切割后的小图不直接给 Sora。它们先被 Canvas 拼成融合图（带编号①②、标题行、角色三视图），再上传 OSS，最后作为 `images` 参数传给 Sora API。

2. Sora 的主要驱动是文字提示词，不是图片。参考图只是辅助——让 Sora "看到"角色外观和场景风格，但视频的实际画面可能和九宫格里的画不完全一样。

3. 视频之间没有帧级连接。上一个视频的最后一帧不会作为下一个视频的输入。跨视频的连续性完全靠文字描述 + 参考图让 AI "猜"着保持一致，这就是为什么称之为"软保证"。

        if (shot.shotSize) sceneParts.push(shot.shotSize);

- 最大尝试次数：240 次
- 实际间隔：5000ms（5 秒），超时 = 240 × 5 秒 = 20 分钟
- 完成状态判断：completed / succeeded / success / done / finished（任一匹配）
- 错误状态判断：error / failed / failure（任一匹配）
- 只有状态完成且有 videoUrl 才返回成功
- 超时后最后查一次，仍无结果则报错
  generateMultipleSoraVideos — 批量串行执行
  for (const tg of taskGroups) {
  try {
  const result = await generateSoraVideo(tg, onProgress, context);
  results.set(tg.id, result);
  } catch (e) {
  results.set(tg.id, { status: 'error', progress: 0 });
  }
  }
  串行执行，单个失败不中断。返回 Map<taskGroupId, SoraVideoResult>。
